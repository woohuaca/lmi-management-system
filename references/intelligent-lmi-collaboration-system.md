# 智能 LMI 协同系统升级草案

这份文档不是在重复 LMI 理论，而是在定义：

- 这套系统现在已经具备哪些指导原则
- 它为什么还没有完全成为“智能协同系统”
- 后续应该按什么结构升级

---

## 一、当前已经存在的指导原则

当前 skill 已经明确了几条核心原则：

1. 目标先于任务  
   日、周、月的计划都不应从零散待办出发，而应从角色价值和目标结果出发。

2. 时间服务目标  
   时间块不是装饰，而是对高权重角色、高回报活动和关键结果的保护。

3. 少数高回报活动优先  
   系统应持续帮助用户从一堆任务里压缩出最值得保护的 1-3 件事。

4. 计划链必须连通  
   默认逻辑应是：`角色 -> 半年/季度方向 -> 月 -> 周 -> 日 -> 复盘 -> 调整`

5. 复盘必须导向调整  
   复盘不是“记录心情”，而是输出继续 / 延后 / 授权 / 删除 / 机制重设计。

6. AI 应该先给草案，再少量追问  
   不用把用户卡在“信息不完整”，而是先起草、标假设、再局部追问。

7. 每天的系统状态必须刷新  
   今日计划不能靠旧聊天上下文，必须先刷新当天文件，再回答和执行。

8. 时间设计要尊重人的恢复规律  
   长时间、高认知负荷、外出、会议密集或切换成本高的任务之后，应主动留出缓冲和恢复时间，而不是把时间表排满。

9. 关键任务要占据最合适的大块时间  
   重要任务不应被塞进碎片时间，而应优先进入精力最好、干扰最少、可持续推进的大块时间。

10. 固定承诺要进入稳定日历  
   已确认的会议、外出、复盘、关键推进块，应该沉入飞书日历或其他稳定日历，而不是只存在于聊天和草案里。

这些原则本身是对的，而且已经构成了系统的骨架。

---

## 二、当前系统还不够“智能”的地方

目前系统更像：

- 一个 LMI 方法库
- 一个计划 / 复盘模板集合
- 一组脚本和提醒器
- 若干针对 OpenClaw / Feishu 的运行补丁

它已经“能工作”，但还没有完全变成“智能协同系统”。

主要差距在 4 点：

1. 原则层、策略层、运行层混在一起  
   `SKILL.md` 里同时放了 LMI 原则、日计划策略、focus 协议、Feishu 运行限制，导致结构越来越重。

2. 系统仍以“文本段落”而不是“工作对象”驱动  
   现在很多判断还是靠解析 markdown 标题和句子，而不是围绕统一对象来决策。

3. 结论缺少“来源”和“理由”  
   为什么是 A1？为什么某件事进 C 不进 A？为什么今天承接的是这个？系统内部知道一部分，但用户看不到。

4. AI 还承担了太多状态责任  
   理想状态下，脚本应该负责状态和规则，模型负责解释和协同。现在有时模型还在补状态、猜来源、修上下文。

---

## 三、目标形态：什么才叫“智能 LMI 协同系统”

目标不是让它“能生成更多内容”，而是让它具备这 5 种能力：

1. 能理解层级  
   知道一个输入是在角色层、月层、周层、日层，还是复盘层。

2. 能理解承接  
   知道今天来自哪一个周目标，本周来自哪一个月重点，本月来自哪一个季度目标。

3. 能理解冲突  
   知道某项工作为什么应该降级、延后、授权，或为什么今天不该进计划。

4. 能解释结果  
   不只输出结论，还能说明“为什么这样排”。

5. 能从复盘中学习  
   不是每次从零开始，而是逐步知道：
   - 哪些 A 项总被拖延
   - 哪些高回报活动真的有效
   - 哪些提醒方式有用
   - 哪些角色边界长期不清晰

---

## 四、建议的三层架构

### 1. 原则层

负责定义长期稳定、不该频繁变动的内容：

- LMI 核心原则
- 计划链路
- 复盘逻辑
- 管理者视角
- 协同风格

这层应该主要存在于：

- `SKILL.md`
- `references/lmi-core.md`
- `references/agent-collaboration-rules.md`

### 2. 策略层

负责定义“系统如何做决策”：

- Q -> 月 -> 周 -> 日 的承接规则
- A / B / C / D 的分类规则
- Inbox 的重决策规则
- 日计划与复盘的压缩规则
- 哪些事项属于高回报活动
- 哪些事项需要授权或机制化

这层应该尽量清晰、显式、可验证，不依赖模型自由发挥。

### 3. 运行层

负责具体执行：

- Feishu 消息发送
- 提醒器
- focus session
- 文件生成
- daily freshness
- 日历连接
- live workspace 同步

这层应当被视为实现细节，不应反向污染上层原则。

---

## 五、建议建立统一对象模型

现在最值得补的一层不是更多模板，而是统一对象。

建议至少定义这些对象：

- `Role`
  - 名称
  - 权重
  - 职责
  - 关键结果
  - 高回报活动

- `Goal`
  - 所属层级：Q / 月 / 周 / 日
  - 来源
  - 归属角色
  - 成功定义
  - 截止时间

- `Thread`
  - 一条持续推进的业务线程
  - 例如：客户转化、Charter 提交、PIC 评审、POC 交付

- `InboxItem`
  - 输入内容
  - 决策状态
  - 去向：进明天 / 留本周 / 转项目事实候选 / 丢弃

- `DailyResult`
  - 今日主结果
  - 关联周目标
  - 关联线程

- `Reminder`
  - 提醒对象
  - 来源计划
  - 状态

一旦对象统一，很多现在靠正则和临时规则解决的问题就会自然收敛。

## 六、建议的协同分工

### 脚本负责什么

- 读取来源文件
- 建立对象
- 做承接判断
- 做显式规则决策
- 生成结构化结果
- 记录状态

### AI 模型负责什么

- 压缩表达
- 补充解释
- 发现层级错位
- 协助重决策
- 把“结构化状态”翻译成自然对话

### 不建议 AI 负责什么

- 用旧上下文替代最新状态
- 直接脑补今天的计划状态
- 自由决定来源优先级
- 在状态不清楚时继续包装成“看起来完整的计划”

一句话说：

`脚本先定状态，模型再做协同。`

---

## 七、建议补上的三个系统能力

### 1. 来源可解释

每个关键结论都应该能回答：

- 它来自哪份文件？
- 它为什么进入 A1？
- 它为什么不是 D 或 C？

这层不一定默认展示给用户，但系统内部必须有。

### 2. 冲突可诊断

系统不仅要输出计划，还要知道：

- 目标和日历是否冲突
- 角色权重和时间投入是否冲突
- 周目标和今日安排是否冲突
- 提醒状态和当日计划是否冲突
- 大任务和恢复时间是否冲突
- 关键任务是否被排进了错误的时间块

### 3. 复盘可学习

系统每周都应能回答：

- 哪些 A 项总被延后
- 哪些线程反复出现但没有收口
- 哪些高回报活动带来真实推进
- 哪些事项应该转机制，而不是继续靠意志力
- 哪些时间块安排最容易成功
- 哪些长任务后实际需要恢复缓冲
- 哪些固定安排应该前置写入日历而不是临时协调

---

## 八、建议补上的时间设计原则

为了让系统更像“协同系统”而不是“排班器”，建议把下面这些时间原则升级成一等规则：

1. 大块时间优先给关键任务  
   A1、最高权重角色任务、高回报活动，应优先进入最适合的 90-180 分钟大块时间。

2. 长任务后必须有弹性  
   深度写作、密集会议、外出、路演、客户拜访之后，不应立即无缝衔接下一个重任务，应预留恢复 / 收口 / 机动时间。

3. 固定事项要沉入日历  
   已确认的会议、出行、评审、复盘、固定节奏动作，应更新到飞书日历或稳定日历源，避免系统永远停留在“假定安排”。

4. 最佳时间要服务最佳任务  
   早晨深度块、下午整块窗口、安静环境、路途中可处理的轻任务，应该与任务类型匹配，而不是平均分配。

5. 计划里要有机动空间  
   每天和每周都应保留一定缓冲，不把所有可用时间填满，否则系统会持续高估执行能力。

---

## 九、升级优先级建议

按价值排序，我建议优先做这 4 件事：

1. 把 `Q -> 月 -> 周 -> 日` 承接链变成系统一等规则  
   不再主要依赖文件名和文本猜测。

2. 把 `Inbox` 从展示文本升级为可决策对象  
   它应该进入决策链，而不是只出现在旁边。

3. 给 `今日计划 / 周计划` 增加来源解释能力  
   先内部可见，再决定对用户展示多少。

4. 把 `SKILL.md` 重构成三层结构  
   原则层稳定，策略层清晰，运行层独立。

---

## 十、判断标准

如果这套系统升级成功，应该出现这些变化：

- 用户不需要反复提醒“这条承接逻辑不是这样的”
- 计划不再主要像信息堆叠，而更像有判断的协作结果
- 系统遇到缺失输入时会诚实暴露，而不是继续脑补
- 日、周、月之间的承接关系更稳定
- AI 更像搭档，少像一个会拼接模板的机器人

---

## 十一、一句话总结

下一阶段的目标，不是把 LMI skill 做得更“会写”，而是把它做得更“会判断、会承接、会解释、会学习”。
