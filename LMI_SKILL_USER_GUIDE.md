# lmi-management-system 使用说明

这份说明面向第一次试用 `lmi-management-system` 的用户。

它不把 Skill 当成一个普通 Todo List，也不把它当成一次性提示词模板。更准确地说，它是一套基于 LMI 方法的 AI 管理协作方式：帮助你把角色、目标、计划、执行和复盘串成一个可以持续调整的闭环。

建议新用户不要一开始就跑完整体系。最容易开始的入口是：**先做日复盘**。

因为日复盘最贴近真实材料：今天做了什么、什么没做完、什么打断了你、明天最担心什么。只要这一步跑起来，就可以自然接到第二天的日计划，再逐步接到周复盘、周计划、月计划和角色复盘。

---

## 1. 先理解 LMI

### 1.1 LMI 解决的不是任务，而是目标和时间投入

LMI 的重点不是“今天列多少任务”，而是：

- 我现在承担什么角色？
- 这些角色要创造什么结果？
- 哪些活动最能推动结果？
- 我的时间是否真的投向了这些高回报活动？
- 复盘之后，下一步行为要怎样调整？

所以，LMI 时间管理不是简单安排日程，而是把时间放回到角色和目标里看。

普通任务工具通常回答：

```text
今天要做什么？
```

LMI 更关心：

```text
为什么今天要做这件事？
它服务哪个角色和目标？
如果没做完，下一步应该继续、延后、授权，还是停止？
```

### 1.2 关键概念

**角色**

角色不是职位名称，而是你当前承担的价值创造责任。

例如：产品负责人、客户机会发现者、项目管理者、团队管理者、个人能力建设者。

**目标**

目标是某个角色在一个周期内要达成的结果。目标要尽量具体，能被复盘，不只是愿望。

**高回报活动**

高回报活动是少数最能推动目标的行动。它们不一定最紧急，但通常最值得保护时间。

比如：

- 深度客户访谈
- 关键方案设计
- 重要客户跟进
- 团队机制建设
- 关键文档或原型产出

**时间投入**

时间花在哪里，实际优先级就在哪里。

如果你的目标很重要，但一周没有真正投入连续时间，它在系统里就应该被视为风险。

**A/B/C/D**

日计划使用 A/B/C/D 分类：

- `A`：重要事项，直接服务角色目标或本周关键结果
- `B`：紧要事项，有时限压力或外部承诺
- `C`：联络 / 跟进事项
- `D`：会议 / 讨论 / 协调事项

分类不是为了贴标签，而是为了决定注意力如何分配。

**复盘**

复盘不是写流水账，而是把经验转成下一步调整。

一个有效复盘至少要回答：

- 今天最大推进是什么？
- 最大打断是什么？
- 未完成事项如何处理？
- 明天第一步是什么？

**明天第一步**

明天第一步是日复盘和第二天日计划之间的连接点。

如果复盘没有生成明天第一步，它很容易停留在总结；有了明天第一步，复盘才会进入下一轮行动。

**机制问题**

有些问题不是靠个人意志硬扛能解决的，而是需要机制。

例如：

- 需要固定检查点
- 需要授权
- 需要减少会议
- 需要明确决策人
- 需要调整团队节奏

管理类复盘里，Skill 会尽量提醒你区分目标问题、执行问题和机制问题。

### 1.3 Skill 的基本闭环

完整闭环是：

```text
角色目标 -> 月计划 -> 周计划 -> 日计划 -> 日复盘 -> 周复盘 -> 月复盘 -> 角色复盘
```

但新用户建议从这个最小闭环开始：

```text
日复盘 -> 明天第一步 -> 日计划 -> 日复盘
```

跑顺之后，再加：

```text
周复盘 -> 周计划 -> 月计划 -> 角色复盘
```

### 1.4 Skill 的设计原则与策略

`lmi-management-system` 的设计重点，不是让 AI 多生成一些计划，而是让 AI 按 LMI 的判断逻辑协同你做取舍。

它有几条核心设计原则。

**1. 目标先于任务**

Skill 会尽量先看角色、月目标和周目标，再处理今天的具体事项。

原因很简单：昨天没做完的事，不一定今天还重要；今天突然冒出来的事，也不一定应该成为今天的 A1。

**2. 压缩优先级，而不是扩展任务清单**

当你给出很多任务时，Skill 不应该只是把它们排得更整齐，而应该帮你压缩重点。

特别是在日计划和周计划里，它会尽量把注意力收敛到少数关键结果和高回报活动上。

**3. 未完成事项必须重新决策**

Skill 不鼓励把未完成事项自动滚到明天。

每个未完成事项都应该重新判断：

- 继续
- 延后
- 授权
- 删除
- 留作跟进

这个策略是为了避免计划越来越臃肿，也避免昨天的心理负债自动占据今天。

**4. 新输入先隔离，再决策**

白天出现的新想法、提醒、风险和跟进事项，不一定要马上进入今天计划。

Skill 会优先建议你把它们收进 Inbox，等日复盘或周复盘时再判断去向。

这样可以保护当天的 A 类事项，减少临时输入对注意力的劫持。

**5. 复盘必须产生调整**

复盘不是为了记录更多内容，而是为了改变下一步行动。

所以 Skill 会特别关注：

- 明天第一步是什么
- 哪些事项要继续、延后、授权或删除
- 哪些问题属于目标问题、执行问题或机制问题

如果复盘没有影响明天，它就还不够有效。

**6. 管理问题要看机制，不只看个人努力**

对于管理者或复杂协作场景，Skill 不会把所有偏差都归因于个人执行力。

它会尽量提醒你检查：

- 是否需要授权
- 是否需要固定节奏
- 是否需要检查点
- 是否需要明确责任人
- 是否需要减少或重构会议

**7. 先起草，再最少追问**

Skill 的协作策略是：在信息不足但方向大致明确时，先给出一版草案，再问最少的问题。

这样做的目的是降低启动成本。很多时候，用户不是没有想法，而是不想从空白页开始。

**8. 让日、周、月、角色互相承接**

Skill 的长期价值来自闭环承接：

- 日计划承接周计划和昨日复盘
- 周计划承接月目标和周复盘
- 月计划承接角色目标和月复盘
- 角色复盘反过来调整角色权重和高回报活动

也就是说，每一层不是孤立模板，而是下一层计划和上一层复盘之间的连接。

## 2. 从日复盘开始使用

### 2.1 为什么建议从日复盘开始

很多人一开始会想先做角色澄清或月计划。

这当然可以，但对新用户来说，最高层的目标往往不够清楚，容易卡住。

日复盘更容易开始，因为你已经有真实材料：

- 今天完成了什么
- 哪些事没完成
- 什么打断了你
- 哪些事项让你焦虑
- 明天最需要先动哪一步

从日复盘开始，Skill 不需要先假设一个完美目标体系，而是从真实工作里反推你的注意力、目标和节奏问题。

### 2.2 日复盘：把今天转成明天第一步

**什么时候用**

每天收工前，或第二天早上补做也可以。

**你可以这样说**

```text
请使用 lmi-management-system 帮我做今天的日复盘。

今天完成了：
1. ...
2. ...

没完成的是：
1. ...
2. ...

最大打断是：
...

明天最担心的是：
...
```

如果你没有整理材料，也可以直接说：

```text
请使用 lmi-management-system 帮我做今天的日复盘。
你先问我最少的问题，帮我整理成复盘。
```

**Skill 会重点看**

- 今天最大推进是什么
- 今天的 A1 是否发生
- 注意力被什么打断
- 未完成事项应该继续、延后、授权还是删除
- 明天第一步是什么

**输出内容通常包括**

- 今日最大推进
- 最大打断
- 工作体验
- 未完成事项去向
- 明天第一步
- Next 1-3 Moves

**关键点**

不要把日复盘写成“今天做了哪些事”的列表。要让它产生一个明确的明天第一步。

### 2.3 日计划：把复盘承接到今天

日复盘之后，第二天自然进入日计划。

**什么时候用**

每天早上，或开始工作前。

**你可以这样说**

```text
请基于昨天的日复盘，帮我排今天。
按 A/B/C/D 分类，并给出今日主结果。
```

如果是在已有协作线程里，建议用更强的说法：

```text
请先刷新今天的 LMI 日计划，再按最新周计划和月目标帮我排今天。
```

**Skill 会重点看**

- 昨日复盘和明天第一步
- 本周目标
- 本月目标
- 今天固定日程
- 未完成事项是否仍然值得继续

**输出内容通常包括**

- 昨日承接
- 今日主结果
- A：重要事项
- B：紧要事项
- C：联络 / 追踪事项
- D：会议 / 讨论 / 协调事项
- 今日安排
- 收工前检查

**关键点**

不要把昨天没完成的事情全部滚到今天。

日计划最重要的判断是：今天的主结果是什么？哪些事项真的值得成为 A？

### 2.4 周复盘：从每天的偏差里看模式

当你做了几天日复盘，就可以做周复盘。

**什么时候用**

周五下午、周末，或下周一开始前。

**你可以这样说**

```text
请使用 lmi-management-system 帮我做本周复盘。
重点看高回报活动、低回报时间、目标问题、执行问题和机制问题。
```

如果你有每日记录，可以一起贴上：

```text
这是我本周每天的日复盘摘录：
...
请帮我压缩成本周复盘，并给出下周调整建议。
```

**Skill 会重点看**

- 本周最重要目标是否推进
- 时间是否投向最高权重角色
- 哪些事项反复顺延
- 打断是否有重复模式
- 哪些问题属于目标、执行或机制问题
- 下周应该保留、调整或停止什么

**输出内容通常包括**

- 本周目标完成情况
- 高回报活动复盘
- 低回报时间 / 打断模式
- 目标问题
- 执行问题
- 机制问题
- 下周调整建议

**关键点**

周复盘不要只看完成率。它更重要的价值是发现模式：

- 哪类任务总是被拖延？
- 哪类会议总是挤压 A 类事项？
- 哪个角色长期拿不到时间？
- 哪些事不该再靠自己硬扛？

### 2.5 周计划：把月目标拆成 1-3 个结果

周复盘之后，就可以做周计划。

**什么时候用**

周一上午，或周末提前做。

**你可以这样说**

```text
请基于本月目标和上周复盘，帮我做本周计划。
压缩出本周最重要的 1-3 个结果，并给出高回报活动和保护时间块。
```

**Skill 会重点看**

- 本月目标
- 上周复盘
- 本周可用时间
- 本周最高权重角色
- 哪些高回报活动需要被保护
- 哪些事项可以授权或只跟进

**输出内容通常包括**

- 本周主题
- 本周主推进角色
- 本周 1-3 个关键结果
- 高回报活动
- 保护时间块
- 授权与跟进
- 周末复盘钩子

**关键点**

周计划不是把所有事项列出来，而是压缩出本周必须形成的结果。

如果本周结果超过 3 个，通常需要再压缩。

### 2.6 月复盘与月计划

月度层面适合在月末和月初使用。

**月复盘适合这样说**

```text
请使用 lmi-management-system 帮我做月复盘。
重点检查本月目标、角色时间投入、高回报活动和机制问题。
```

**月计划适合这样说**

```text
请使用 lmi-management-system 帮我做本月计划。
先整理本月主角色、个人焦点目标、公司焦点目标，再收敛首要目标和高回报活动。
```

**月复盘看什么**

- 本月目标是否完成
- 哪些角色时间投入不足
- 高回报活动是否被保护
- 哪些目标应该延后或停止
- 哪些机制需要调整

**月计划看什么**

- 本月主角色
- 本月焦点目标
- 公司目标 / 个人目标
- 高回报活动
- 资源、风险和节奏安排

**关键点**

月计划要避免写成“大愿望”。它要能拆到周计划。

月复盘要避免只总结结果。它要能影响下个月的角色权重、时间投入和机制安排。

### 2.7 角色复盘与角色澄清

角色层面最高，建议月度或季度做一次。

**什么时候用**

- 感觉自己很忙但方向不清
- 职责变化
- 项目阶段变化
- 时间投入和目标长期不匹配
- 准备季度计划或年度计划

**角色复盘可以这样说**

```text
请使用 lmi-management-system 帮我做角色复盘。
检查我的职责、关键业绩、时间投入和角色边界是否仍然匹配。
```

**角色澄清可以这样说**

```text
请使用 lmi-management-system 帮我梳理角色目标澄清表。
我的当前职责包括：...
我的主要项目包括：...
我希望你帮我识别角色、目标、关键结果和高回报活动。
```

**Skill 会重点看**

- 当前角色是否仍然准确
- 角色权重是否要调整
- 每个角色的关键结果是否清晰
- 哪些事情应该授权、停止或机制化
- 哪些高回报活动需要进入月计划和周计划

**关键点**

角色澄清不是写岗位说明书，而是判断你最应该创造什么价值。

### 2.8 Inbox：白天先收集，晚上再决策

Inbox 用来处理临时出现的想法、提醒、风险和跟进事项。

它的目的不是收集更多，而是避免突发事项马上污染今天计划。

**白天可以这样说**

```text
收进 Inbox：我想到一个客户洞察模板，月底再决定要不要升级。
```

```text
这个先别进今天，收进 Inbox：和某位同事约原型评审。
```

**晚上可以这样说**

```text
先做日复盘，再清 Inbox。
帮我判断哪些进明天、哪些留在本周、哪些转项目事实候选、哪些丢弃。
```

**Inbox 决策只有四类**

- 进明天
- 留在本周
- 转项目事实候选
- 丢弃 / 仅记录

**关键点**

Inbox 不应该变成另一个永久待办清单。

它白天负责收集，晚上或周复盘时必须重新决策。

### 2.9 Focus：给关键事项留下时间证据

Focus 用于 A 类事项、关键 B 类事项和高回报活动。

它的价值是记录真实时间投入，方便日复盘、周复盘和角色复盘。

**开始专注**

```text
开始 A1 的 50 分钟专注。
```

或：

```text
开始 90 分钟深度块，任务是完成深度调研报告第一版框架。
```

**结束专注**

```text
结束本轮专注，记录结果：完成目录和判断框架，focus score 4。
```

**复盘专注投入**

```text
今天按角色看，我的专注时间投在哪里了？
```

```text
这周哪个角色拿到了最多深度时间？
```

**关键点**

不要把所有事情都开专注。

Focus 主要用于 A 类事项、关键 B 类事项和高回报活动。C / D 类事项通常更适合批量处理。

## 3. 让 Skill 变得更适合你

### 3.1 先让它理解你的工作上下文

Skill 越了解你的角色、目标和节奏，输出越容易贴近真实工作。

建议一开始给它这些信息：

- 当前角色
- 本月目标
- 当前项目
- 常见会议
- 固定节奏
- 不希望被打断的时间
- 长期偏好
- 常见协作对象

可以这样说：

```text
以后请使用 lmi-management-system 协同我。

我的主要角色是：
1. ...
2. ...

我本月最重要目标是：
...

我希望每天计划不要超过 3 个 A 类事项。
如果我给出新想法，请优先建议是否收进 Inbox。
如果是管理问题，请提醒我检查授权、节奏和机制。
```

### 3.2 用反馈持续校准输出

不要只被动接受 Skill 的输出。

如果它排得太满、太散、太模板化，要直接反馈。

常见反馈方式：

```text
这个日计划太满，请压缩到 1 个 A1 和 2 个 B 类事项。
```

```text
这个复盘太像流水账，请帮我区分目标问题、执行问题和机制问题。
```

```text
今天 A1 不应该是这个，它没有服务本月目标，请重新判断。
```

```text
这些 C 类跟进可以合并，不要拆成太多条。
```

```text
这个建议太理想化，请按我今天只有 2 小时深度时间重新安排。
```

**关键点**

LMI Skill 的价值不是一次就生成完美计划，而是在反馈中逐渐学习你的判断方式。

### 3.3 把反复有效的做法固化成规则

如果你发现某些反馈经常重复，就可以把它变成长期规则。

例如：

- 每天只保留一个 A1
- 上午优先放高回报活动
- 临时想法默认进 Inbox
- 未完成事项必须重新决策
- 周五固定做周复盘
- 日复盘必须产出明天第一步
- 管理事项要检查授权和机制

可以写成长期指令：

```text
以后做 LMI 日计划时，请默认：
1. 先看月目标和本周目标，再看昨天未完成事项；
2. 每天只保留 1 个 A1；
3. 新想法不要直接进入今天，先建议是否收进 Inbox；
4. 日复盘必须输出明天第一步；
5. 如果是管理问题，请检查是否需要授权、节奏或机制；
6. 结尾只给 Next 1-3 Moves。
```

### 3.4 Codex 和小龙虾如何分工

如果你同时使用 Codex 和小龙虾，可以这样分工。

**小龙虾 / 日常助手**

适合负责日常陪跑：

- 每天排计划
- 做日复盘
- 收 Inbox
- 启动专注块
- 做周复盘 / 月复盘
- 在执行中提醒你回到 A1

**Codex**

适合负责系统打磨：

- 修改 Skill
- 调整模板
- 增加脚本
- 检查版本
- 写使用说明
- 把反复有效的做法固化成规则
- 根据真实使用反馈优化协作方式

一句话：

```text
小龙虾负责日常陪跑，Codex 负责把反复有效的做法固化进 Skill。
```

### 3.5 推荐长期指令

你可以在新的协作线程里先发这一段：

```text
以后请优先使用 lmi-management-system 协同我。
默认先判断我是角色目标澄清、月计划、周计划、日计划、日复盘、周复盘、月复盘、角色复盘、时间图像还是统计分析场景。
尽量先起草，再最少追问。
优先压缩重点，默认检查高回报活动、授权事项和管理机制。
如果我给出新想法、提醒或突发事项，先建议是否收进 Inbox。
如果我确认了 A1 / A2，请主动建议是否开启 25 / 50 / 90 分钟专注块。
每次结尾输出 Next 1-3 Moves。
```

如果你已经有明确的个人偏好，可以继续补充：

```text
我的个人偏好：
1. 每天只保留 1 个 A1；
2. C 类跟进尽量合并；
3. 不要把未完成事项自动滚到明天，必须重新决策；
4. 日复盘必须产出明天第一步；
5. 周复盘必须区分目标问题、执行问题和机制问题。
```

## 4. 一个推荐的试用路径

如果你是第一次使用，建议按 7 天试用。

**第 1 天：做一次日复盘**

```text
请使用 lmi-management-system 帮我做今天的日复盘。
你先问我最少的问题，帮我整理出今日最大推进、最大打断和明天第一步。
```

**第 2 天：基于昨天复盘做日计划**

```text
请基于昨天日复盘，帮我排今天。
按 A/B/C/D 分类，并给出今日主结果。
```

**第 3-4 天：开始使用 Inbox**

```text
收进 Inbox：...
```

晚上：

```text
先做日复盘，再清 Inbox。
```

**第 5 天：为 A1 开一次专注块**

```text
开始 A1 的 50 分钟专注。
```

结束后：

```text
结束本轮专注，记录结果：...
```

**第 6 天：整理本周记录**

```text
请帮我把这几天的日复盘压缩成一份周复盘草案。
重点看高回报活动、打断模式和下周调整。
```

**第 7 天：做下周计划**

```text
请基于本周复盘，帮我做下周计划。
压缩出最重要的 1-3 个结果，并给出高回报活动和保护时间块。
```

---

## 5. 如何判断这次试用是否有效

试用一周后，可以用这些问题判断：

- 我是否更清楚今天的 A1？
- 我是否减少了未完成事项自动滚动？
- 我是否知道哪些突发事项应该先进 Inbox？
- 日复盘是否真的生成了明天第一步？
- 周复盘是否发现了重复打断或机制问题？
- 我的时间是否更接近角色目标和高回报活动？

如果这些问题中有 2-3 个开始改善，说明 Skill 已经开始发挥作用。

如果没有改善，通常不是不能用，而是需要继续校准：

- 提供更清楚的角色目标
- 压缩 A 类事项
- 明确你的长期偏好
- 把不合适的输出反馈给 Codex 或小龙虾
- 把反复有效的做法固化成规则

---

## 6. 最后提醒

`lmi-management-system` 不是为了让你管理更多任务。

它的目标是帮助你持续判断：

```text
我的时间是否服务了真正重要的角色和目标？
```

如果只把它当成 Todo List，它会显得有点重。

如果把它当成一个能陪你做计划、复盘、校准和调整的管理协作者，它的价值会慢慢显现出来。
